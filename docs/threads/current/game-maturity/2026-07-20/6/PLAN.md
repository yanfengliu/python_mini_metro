## Canonical fleet and action contract

- `Mediator.num_metros` remains writable total capacity. `Mediator.metros` remains the assigned-only global list. `Path.metros` owns route assignment. `available_locomotives` remains `max(0, num_metros - len(metros))`; queued locomotives remain assigned until physical detachment, so no early refund or third resource pool exists.
- `PathLifecycle.finish_path_creation` finishes topology, creation flags, and path-button assignment without resolving or constructing a Metro. A new line is valid and unserved until the player assigns one. Existing creation rollback/partial-button behavior remains otherwise exact.
- Add public `Mediator.assign_locomotive(path) -> bool` and `Mediator.queue_locomotive_unassignment(path) -> bool` facade methods backed by one dependency-light stateless `FleetManagement` component. The component retains no facade, path, entity, collection, factory, or callback between calls.
- Add strict structured actions `{"type": "assign_locomotive", "path_id"|"path_index": ...}` and `{"type": "unassign_locomotive", "path_id"|"path_index": ...}`. Exactly one selector is required. IDs must be nonempty strings resolving to exactly one active path; indices must be real in-range integers, never booleans. Unknown, ambiguous, duplicate-ID, detached, creating, malformed, terminal, or otherwise inapplicable targets reject without time, RNG, UUID, collection, pointer, or observation effects.
- Assignment validates the exact active path, nonempty availability, completed usable geometry, unique ownership, and current collections before resolving the Metro factory. It constructs one fresh Metro, invokes the canonical `Path.add_metro` initialization, and installs the exact same identity into the current global list. Factory, route-add, global-append, append-then-raise, rebound-list, or postcondition failure restores the original path/global collection objects and contents and leaves no partially assigned live entity.
- Multiple locomotives per path are supported. New identities append deterministically to `path.metros` and global `metros`; assignment does not reorder or mutate existing entities. Artificial over-cap states reject assignment until the clamped deficit clears.

## Queued unassignment and redistribution

- One unassignment request selects the last empty, nonqueued Metro in reverse `path.metros` order through one shared pure eligibility query used by both mutation and presentation. If no such exact globally owned Metro exists, the request rejects. GM-06b deliberately does not accept an already occupied candidate; GM-06d owns occupied-train queue semantics.
- Add one exact boolean `Metro.is_unassignment_queued`, default false. A queued moving Metro remains in both collections, keeps its route/identity, counts assigned, cannot board, and is forced to stop at its next real station. `Mediator.can_board_at_station` is false and `Mediator.get_boarding_candidates_for_metro` returns an empty candidate sequence for that exact Metro in both mutating and nonmutating modes, so eligible waiting riders remain at the station and no boarding dwell begins. It continues ordinary movement and applicable unloading; direct malformed rider injection never authorizes destructive removal.
- A Metro is at a real station only when `metro.current_station` is the exact identity of a station in its owning path; endpoint-coordinate equality alone is insufficient. Consequently a freshly assigned Metro whose `Path.add_metro` position is at the segment start but whose `current_station` is `None` queues and must travel to its first actual stop instead of detaching immediately. If the selected empty Metro is already at a real station, the accepted request detaches it transactionally in the same action.
- Later settlement runs after the complete `PassengerFlow.increment_time` call returns, including its same-tick game-over evaluation. An empty queued Metro that arrives on a terminal-producing tick therefore still settles once before the tick returns; only subsequent terminal ticks are frozen. Settlement detaches exactly once only when the Metro is empty, at a real station by identity, globally present once, and owned by exactly one active path. Removal failure or malformed ownership restores source/global collection objects and contents and leaves the request pending rather than inventing availability.
- Paused play accepts a request: immediate safe station detachment may complete, while a moving queued Metro cannot advance. Terminal structured and manual control paths reject. Route replacement/redraw/handle edits preserve pending identity and state across success/rejection/rollback. Removing a line may remove its empty queued Metro through the existing line-removal boundary; occupied/destructive line-removal behavior remains explicit GM-06d debt.
- Repeated requests may queue distinct empty Metros on one path. Duplicate requests cannot target an already queued Metro and rejected requests never cancel a prior queue.
- Redistribution is intentionally compositional: queue a source locomotive, wait until the canonical availability count returns, then assign on the destination. Manual plus/minus controls and the two structured actions expose the same sequence. GM-06b adds no compound target reservation, destination queue, same-tick teleport, or `redistribute_locomotive` action because each would introduce target-staleness/cancellation state or a third ownership category reserved for later review.

## Player-visible controls and rendering

The control resolves the path through the currently assigned path button at release time, so button reassignment cannot leave stale ownership. For every control in both registered fast and fidelity profiles, `map_action_coordinate(*canonical_to_action_coordinate(center, profile), profile)` must land inside that exact control and outside its sibling, parent path button, and every station. - Fleet release follows the existing release-target rather than press-origin rule. Every invoked fleet release clears selected-path/redraw and all button hover state in `finally`, whether the facade succeeds, returns false, or raises; exceptions propagate after cleanup.
