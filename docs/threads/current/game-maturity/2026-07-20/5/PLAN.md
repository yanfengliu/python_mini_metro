## Canonical inventory model

- `Mediator.num_metros` remains the writable total/cap compatibility surface. `Mediator.metros` remains the assigned-only live global collection used by simulation, rendering, observations, path replacement, and passenger flow. Unassigned inventory is fungible numeric capacity, not preconstructed `Metro` objects.
- Add one read-only late-derived `available_locomotives` property: `max(0, num_metros - len(metros))`. It owns no cache or backing field and re-reads both public compatibility surfaces on every access. In a valid gameplay-reachable state, `available_locomotives + len(metros) == num_metros`, every global Metro identity is unique, and the global set equals the union of exactly one active path owner per Metro.
- Direct legacy writes to `num_metros` remain observable immediately. Increasing the cap permits later automatic allocation only after total exceeds the current assigned count; lowering it never ejects or moves an existing Metro. An artificial over-cap direct mutation reports zero available and blocks later automatic allocation rather than returning a negative count or changing current entities. Exact cap-recovery transitions are `0/2/0 -> 1/2/0 -> 2/2/0 -> 3/2/1` for total/assigned/available. GM-06a does not validate or normalize arbitrary direct state.
- Fresh/reset state is total four, assigned zero, available four, with no unassigned `Metro` construction or UUID consumption. Completing a valid line still automatically allocates exactly one new Metro when availability is positive so the game remains playable until GM-06b supplies explicit controls. At zero availability, the line still completes unserved and the late Metro factory is never resolved, exactly as at baseline.
- Removing a line decreases assigned count by one for each route Metro actually removed from the current global list. In a valid non-over-cap state, each such removal raises availability one-for-one. In an artificial over-cap state, the clamped formula may keep availability at zero while removals instead clear the assignment deficit, including `0/2/0 -> 0/1/0 -> 0/0/0`; no nonexistent resource is reported as returned. A route Metro absent from the global list changes neither assigned nor available; late/rebound collection mutation and the intentionally detached old `path.metros` graph retain their frozen semantics. Current onboard-rider destruction during line removal is documented debt owned by GM-06d and is not falsely claimed fixed here.
- Atomic replacement, full redraw, handle editing, rejected actions, pause/terminal no-ops, and failed/rolled-back GM-05a transactions preserve total and the exact assigned identities; the derived available value therefore remains exact without adding inventory snapshot fields.

## Failure and ordering compatibility

- Preserve path-finish ordering: clear creation flags and temporary geometry, resolve the late Metro factory only when below cap, add the new Metro to the route, append the exact same identity to the current global list, clear `path_being_created`, then dynamically resolve button assignment. Availability changes only when the global append actually changes the assigned collection.
- A factory failure, route-add failure before global append, or global append failure before mutation leaves availability unchanged. An append-then-raise or later button-assignment failure leaves the actually installed Metro assigned and availability consumed. GM-06a adds no counter rollback that would contradict the live collection after a partial effect.
- Path removal remains nontransactional. A failure before any global removal leaves assigned count unchanged; a failure after one or more actual removals reduces assigned count by exactly that number even if later passenger, plan, color, path, button, or replanning work fails. Availability is always recomputed by the clamped formula, so it rises by that number only outside an over-cap deficit. New tests observe the derived invariant rather than changing those established failure seams.
- Reads of `available_locomotives`, HUD drawing, structured observation, and checkpoint comparison are observational: they do not mutate collections, IDs, RNG, clocks, rewards, renderer caches, or topology.
