This unit adds one atomic programmatic route-edit transaction. It does not add selected-line rendering, mouse redraw, endpoint handles, fleet inventory, maps, persistence, or new pixel-policy controls; those remain GM-05b/GM-05c or later work.

The public action is `{"type":"replace_path", exactly one selector, "stations":[...], "loop":bool}`. Top-level dict subclasses and action-type string subclasses retain the existing `isinstance` dispatch contract, while new payload fields are exact types: `path_index` is a non-boolean exact integer, `path_id` is an exact nonempty string, `stations` is an exact list of exact non-boolean active-station indices with at least two entries, and the explicitly present `loop` is an exact boolean. Selector presence is XOR even when a present value is `None`; unrelated extra keys remain tolerated consistently with the existing structured-action surface.

`Mediator` retains real public `replace_path(path: Path, station_indices: List[int], loop: bool = False) -> bool`, `replace_path_by_id(path_id: str, station_indices: List[int], loop: bool = False) -> bool`, and `replace_path_by_index(path_index: int, station_indices: List[int], loop: bool = False) -> bool` methods. Selector wrappers resolve exactly one live completed path and dynamically call the current public `replace_path` hook. Duplicate live path IDs, duplicate target object membership, and selectors of the wrong exact type fail closed instead of silently selecting the first object.

## Topology contract

The stored station order contains unique active station objects as well as unique indices. For loop compatibility only, one terminal repetition of the first index is normalized away; every other duplicate or distinct index aliasing the same Station object is rejected. The loop is represented only by the explicit boolean, and two-station loops remain supported because current path creation supports them.

Replacement is rejected while any manual path creation is active or when the target is not one exact completed member of the live path collection. After normalization, the read-only metro/holder ownership, live segment-reference, position, and onboard-plan invariants run before an exact station-order/loop no-op may return `True`. The valid no-op precedes every factory, geometry, route-planning, RNG, identity, or collection effect; an inconsistent no-op returns `False`, preventing `MiniMetroEnv` from advancing an invalid simulation.

Candidate geometry is built off-live through a `Path` factory resolved only after validation and no-op detection. The candidate receives the target color, path order, normalized active station objects, and loop flag but no metros. After topology installation, the current graph builder resolves once and supplies both marker Nodes and one shared current graph for a stable captured waiting-rider batch. Each scoped rider plan resolves the current `TravelPlan`, search, reducer, and route-selection collaborators only at its effect point. The candidate factory receives no host and is required to be effect-free with respect to the facade; a patched factory that mutates unrelated external state is outside the transaction guarantee, while any candidate-factory exception occurs before live mutation. `PathLifecycle` and its transaction helper remain stateless and domain-import-free through resolver thunks.

The scoped planning boundary is exact and deliberately private: `Mediator._replan_passenger_at_station(passenger: Passenger, station: Station, station_nodes_dict: Dict[Station, Node]) -> None` forwards to the new dependency-light `PassengerFlow.replan_passenger_at_station(...)` operation. Both `PathLifecycle.replace_path(...)` and `PassengerFlow.move_passengers(...)` receive a late-bound `get_scoped_replanner` resolver and call `get_scoped_replanner()(passenger, station, station_nodes_dict)` only at the rider's effect point; `Mediator` supplies `lambda: self._replan_passenger_at_station`. This adds no public Mediator method and no host-protocol callback. Tests rebind the private Mediator hook after construction and prove both callers dynamically dispatch through it, while direct PassengerFlow tests cover the operation itself.

The live path never calls `add_station`, `set_loop`, `remove_loop`, `update_segments`, or `rebuild_geometry` during replacement because their numeric segment-index rebinding can teleport a metro onto a different edge.

Commit preserves the exact live `Path` object, path ID, color, `path_order`, button/color mappings, metro list object, metros, riders, and public collection position. It replaces the contents of the live station/segment/path-segment/padding-segment lists in place and updates only `is_looped` after all preflight checks succeed.

## Metro continuity contract

Preflight validates the entire live network because every onboard plan is marked. The identity union of all `path.metros` and `host.metros` must agree exactly: every live metro occurs once globally, once on exactly one live path, and declares that owning path's ID. Every active passenger occurs exactly once globally and in exactly one station or metro holder, and no two passenger mappings share the same TravelPlan object. A path-only/global-only metro, duplicate holder/global entry, orphan rider, or plan alias rejects without effects.

Every target-line metro must additionally have a current segment whose exact object and index agree with the old segment list. Every moving target-line metro must lie on its old segment: a nonzero segment requires projection within `[0, 1]` and perpendicular residual at most the fixed `1e-6` logical-pixel tolerance, while a zero-length segment requires exact endpoint equality. A failure in any metro or rider makes the multi-metro edit a rejected zero-effect transaction.

A moving metro on a logical path segment maps by the same oriented physical station trip and exact segment endpoints, never by numeric index. A reversed candidate edge flips only the storage-direction flag needed to preserve the same physical target. Position, speed, heading, dwell state, boarding progress, current-station state, and riders remain unchanged.

A moving metro on padding maps only when the same oriented incoming-station, junction-station, outgoing-station transition and exact source/target points exist in the candidate. Loop wrap padding is explicit. If either incident edge changes after the metro has entered padding, the edit is rejected.

Every moving path, moving padding, and stopped-arrival binding enumerates all candidate `(segment_index, is_forward)` states. If more than one state matches, the old motion direction may disambiguate only when it leaves exactly one candidate; otherwise the transaction rejects and never chooses by candidate list order. For stopped metros the preference uses the uniquely reconstructed arrival direction rather than the already-advanced current direction. Equivalent two-station-loop reorderings are therefore allowed only when every metro binding is unique under this rule; unresolved overlap intentionally rejects.

A stopped metro is rebound from its unique oriented arrival edge rather than assuming its already-advanced `current_segment` is the arrival edge. The old arrival edge is reconstructed by enumerating path-segment arrival states and applying the production segment-advance state machine; the candidate must retain that physical arrival edge and endpoint. Advancing the corresponding candidate arrival state yields the post-edit segment/index/direction, allowing an outgoing reroute or new terminus without moving the train. Ambiguous mappings reject under the all-candidate rule above.

Only `current_segment`, `current_segment_idx`, and `is_forward` may change on a successfully mapped metro. No position projection or snap is permitted.

## Passenger transition contract

All station-held passengers are captured in stable station/list order before mutation and synchronously replanned one rider at a time against one post-commit graph after candidate topology is installed. This is global rather than target-plan-only because an added or reordered line can create a shorter route for a passenger whose old plan did not use that line. A new dependency-light scoped passenger-flow operation applies exactly one arrival, route, or fresh fallback disposition without iterating a live holder list, so an arrival removal cannot skip the adjacent captured rider. The existing bulk iterator and its deliberate live-list semantics remain unchanged.

Every onboard passenger across the network is marked to receive a complete fresh route at its next existing safe alight, so route optimality does not depend on whether its old node path happened to mention the edited line. Preflight never calls mutating `TravelPlan.get_next_station()`: it requires an exact non-boolean integer cursor in range, resolves the raw node at that cursor, requires `next_station` to be that node's exact active Station object, requires `next_path` to be the exact owning live line, and requires the target station to occur by identity on that line. Riders on the edited line must retain that target in the candidate topology; otherwise the edit is rejected rather than stranding or teleporting them.

The deferred refresh uses no new hidden field or checkpoint schema. After topology installation, one fresh current-graph build supplies the exact current Node for every target alight; the old topology-derived Node is never reused. The existing plan is reduced to a one-alight marker containing only that fresh node, the cursor resets to zero, `next_station` is that station, and `next_path` remains the rider's current line. The marker remains board-valid and unloads through the existing target-station rule. Its historical route tail is gone, and tests prove cursor/next-station agreement plus exact fresh current-graph path membership because the existing reference oracle checks only index ranges.

At a transfer unload, `PassengerFlow` first moves the rider normally. If incrementing the plan exhausts that one-alight marker, it resets all four plan fields to the existing empty-plan sentinel state before another getter can observe the exhausted cursor, then invokes the new scoped single-rider planner against the graph already built for that passenger-exchange phase. Ordinary non-exhausted plans retain the existing increment-and-next-path behavior. Destination delivery remains first and consumes no marker-replan RNG. A full station blocks the transfer and leaves the marker, cursor, and rider untouched; existing boarding-candidate evaluation may still consume Python RNG before that capacity guard, so the contract is zero invocation of the new marker replanner and exact RNG parity with an otherwise equivalent control run, not an unchanged total RNG state.

The scoped alight planner never invokes the global bulk hook. One successfully transferred marker consumes only its own destination shuffle and cannot re-evaluate unrelated fallback waiters or mutate unrelated arrivals; two markers alighting in one exchange consume exactly two scoped shuffles in metro/rider processing order. The global waiting refresh occurs once during replacement, with exactly one destination shuffle per captured waiter in stable station/list order. Successful seeded twins must reach the same expected Python RNG state, while NumPy remains unchanged. A capacity-blocked marker invokes no scoped planner; its total Python RNG assertion uses exact parity with a control that includes the pre-guard legacy boarding-candidate side effects, and NumPy remains control-identical.

Checkpoint v2 already exposes the marker's line, target station, cursor, single node, holder location, and all metro/topology/RNG state, so no recursive schema change is introduced. Direct tests additionally prove exact path/metro/passenger object and public-ID preservation because canonical checkpoints intentionally erase UUIDs.

## Atomicity and rollback

Expected malformed, ambiguous, unsafe, or inconsistent edits return `False` before live mutation. Effect-free candidate construction failures also occur off-live and propagate without a live rollback. Once the first live field changes, any `BaseException` from transaction topology writes, fresh graph construction, metro/plan effects, or the standard scoped-replanning pipeline restores every field that those declared operations may mutate and re-raises the original failure. Arbitrary side effects introduced by a deliberately rebound factory or hook outside that declared effect surface are not claimed recoverable.

The rollback snapshot records the original host attribute references needed to reverse covered collection rebindings; the collection objects and ordered contents for stations, paths, metros, passengers, and travel plans; the target path scalar fields plus each topology-list object and contents; the three mutable binding fields for every target metro; every pre-existing plan's scalar fields and node-list object/contents; every active station passenger-list object/contents; every metro passenger-list object/contents; every reachable passenger's destination flag, wait value, position reference and coordinates; and the Python RNG state plus a deep copy of the NumPy bit-generator state before the first live effect. If a failing standard pipeline rebinds a covered host collection, rollback reattaches the original object before restoring its contents.

Path buttons and their assigned paths, color ownership maps, progression, clocks, spawning counters, deliveries, line credits, and station visual effects are not standard replacement/replanning effects and must remain identical on ordinary success or preflight rejection. On success, logical and static rendered route geometry must equal the candidate topology while the metro pose remains continuous; on rejection, no-op, or rollback, the prior geometry remains identical. Renderer-owned interpolation/cache state is untouched. Tests snapshot these boundaries so an accidental widened implementation effect cannot pass silently; the rollback fault-injection claim remains limited to the declared standard pipeline above.
