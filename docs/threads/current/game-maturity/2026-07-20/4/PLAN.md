## Two-phase activation and compatibility

- The existing converted primary-pointer down on an assigned unlocked path button continues to arm one ordinary GM-05b `PathRedrawGesture`. Held motion never selects a handle. A valid station release retains full redraw; a zero-station path/speed-button release retains the complete deletion/purchase/speed matrix; captured-draft button release retains cancellation. Releasing anywhere inside the viewport that contains no station or button is the one new non-committing terminal action: regardless of sampled intermediate stations or draft invalidity, it discards the off-live redraw, clears hover state, and latches a transient exact-path weak selection whose handles remain visible while the pointer is up. An off-viewport release is routed by the desktop adapter as one canonical outside-position mouse-up instead of being dropped, and it clears pointer/redraw state without selection or a release-target action. Final-target semantics therefore make dense desktop sampling and PlayerPixel's destination jump agree on selection intent.
- A fresh primary-pointer down that begins inside a visible selected-line handle starts one `PathHandleEdit`, clears the idle selection, and consumes that down/up sequence. Motion then updates only the candidate preview and release completes or cancels the handle operation. A down outside every handle clears idle selection before delegating ordinary station creation, button redraw, or empty-space behavior. Motion crossing a handle can never activate it, so dense desktop sampling and PlayerPixel's destination jump retain identical intent.
- Plain down/up on an assigned path button still deletes because release targets the button rather than empty space. A user therefore selects handles by holding the assigned button, dragging into empty space, and releasing, then begins the edit with a distinct down on the desired handle. Selection clears on operation start/completion/cancel, a new ordinary down elsewhere, reset, or loss of the exact selected path. A false-to-true game-over transition synchronously clears active redraw/handle state, idle selection, pointer-down, and button hover before gameplay input is disabled, releasing every strong edit reference. It is never a saved gameplay mode.
- A finite nondegenerate linear route exposes two filled endpoint handles whose logical anchors extend outward from its visual-lane endpoints. Each finite nondegenerate physical station-to-station edge exposes one hollow insertion handle; loops include the closing edge but no endpoint handles. The exactly coincident reverse edges of a two-station loop collapse to one canonical physical handle whose fixed slot inserts between stored stations zero and one; full redraw remains available for the reverse-equivalent ordering. Degenerate edges omit only an unusable operation.
- One canonical config-driven pure builder owns both render and hit geometry. It derives anchors from centered production lane order, then uses a deterministic logical-ordinal lattice search to relocate centers and leader endpoints away from viewport margins, HUD/control exclusion regions, station envelopes, a conservative config-owned action-quantization margin, and earlier handle envelopes. Same-route crossings and short edges therefore receive distinct accessible centers. Fast/fidelity tests prove that margin covers every registered profile without importing RL policy code into core UI geometry. If adversarial geometry leaves no collision-free viewport lattice point, that handle is omitted fail closed and full redraw remains the fallback.
- The builder returns immutable primitive-only descriptors. Input and rendering recompute/consume those exact canonical descriptors from current state; injected nondefault network-renderer styles do not redefine interactive hit geometry. Descriptors, the idle selection, and render artifacts retain no strong Path, Station, gesture, facade, or other gameplay reference. A nondefault-style test pins hit/render parity at the canonical descriptor even when the injected static renderer uses another lane spacing.
- Hit testing runs only on a fresh down while idle selection is active and chooses the unique nearest descriptor within its visible hit-envelope radius. A nearest-distance tie creates a consumed invalid handle gesture, so its eventual release cannot fall into the historical bare-up deletion/purchase/speed matrix. Selected-mode handles have precedence over station shapes only for that fresh handle down; collision relocation and marker-above-entity rendering keep the intended target visible, while ordinary station-first behavior remains exact whenever no idle handle selection exists.
- `convert_pygame_event` continues to discard the physical pygame button number. Tests still send `button=1`; adding button identity or a new low-level action remains outside GM-05c.

## Edit derivation and atomic commit

- The idle `PathEditSelection` holds only an exact weak path reference; handle descriptors are rebuilt from current live state and remain primitive-only. Starting a handle captures the exact strong path identity, exact source station objects and order, source loop state, handle kind, and insertion slot in one immutable `PathHandleEdit`. It never changes the live path. Finish requires the same path to remain exactly once in the active path collection, the source topology and loop bit to remain identity-exact, and the release station to resolve exactly once in the active station collection; removal, topology mutation, aliasing, or malformed handle/source state cancels fail closed. Button reassignment is deliberately irrelevant because the operation targets the exact selected path, not its current control slot.
- Releasing a linear start endpoint handle on a station absent from the source route prepends it; releasing the end handle on such a station appends it. Releasing on the adjacent interior station removes exactly that one endpoint only when at least two stations remain. Releasing on the original endpoint is a neutral cancel, and releasing on any other existing route station is invalid. Endpoint handles never create, open, or edit a loop.
- Releasing an insertion handle on a station absent from the source route inserts it between that handle's adjacent source stations. The closing-edge handle inserts after the last stored station and preserves loop closure to the first. Any existing source-route station is invalid for insertion.
- Off-station, button-targeted, ambiguous-handle, missing-source, malformed handle/source, same-route, and too-short results cancel with no public hook. Invalid hover receives deterministic red feedback. A locally valid release clears every gesture, idle selection, handle, preview, and button-hover reference before calling the existing public `Mediator.replace_path(path, station_indices, loop)` hook exactly once. Broader canonical path/metro/passenger malformation remains GM-05a preflight responsibility and may therefore receive that one call and return false; GM-05c does not duplicate or extend the already 495-line replacement module.
- The input layer guarantees gesture and hover cleanup if the canonical hook returns false or raises. As in GM-05b, an arbitrary rebound hook is outside the canonical rollback surface, so no claim is made that input cleanup reverses unrelated mutations performed by that hook.

## Preview and rendering contract

An invalid target changes the active handle, candidate preview, and selected button feedback to the existing invalid red. The handle renderer is cache-free and consumes only primitive descriptors.

The live line stays operational underneath until commit; selection and pointer motion never rebuild its topology or static network cache.

Linear start/end, linear interior, loop interior, and loop-closing previews use the same centered lane geometry and two-pass preview rasterizer as a committed candidate.

Handle drawing, removal feedback, candidate construction, and hit testing are observational and leave paths, stations, segments, selections, gestures, renderer inputs, clocks, RNG, snap blips, checkpoints, and structured arrays unchanged. Static and preview caches remain separately bounded at one entry; the cache-free handle renderer, primitive descriptors, and weak idle selection retain no gameplay objects.
