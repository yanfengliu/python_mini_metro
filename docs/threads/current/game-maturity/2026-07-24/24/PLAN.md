# GM-09f plan — map-selection integration (proposed SPLIT; this unit = save-schema v2)

## Scope decision: SPLIT (like GM-09a → GM-09a/a2)
The roadmap's "GM-09f: Menu/save/high-score/RL integration" bundles THREE interdependent but separable concerns, only one of which is high-risk. Per the fleet constitution's "minimal coherent verified unit", split into:
- **GM-09f (this unit): the save-schema v2 MAP FIELD** — the high-risk enabling core. Lets a non-Classic game be saved/loaded; testable end-to-end via the programmatic path (`Mediator(map_definition=RIVER)` → serialize → load), independent of the menu.
- **GM-09f2: the in-game MAP-SELECT menu** — a `MAP_SELECT` `AppScreen` mirroring `_open_settings`/`_handle_settings`, threading the chosen `MapDefinition` into `main.build_game`'s `Mediator(...)`. UI-only; append-only title-rect discipline.
- **GM-09f3: high-score `mapDefinitionVersion`** — bump `HIGHSCORES_SCHEMA_VERSION` to 2, add the entry key, read the map identity from the live mediator in `main.record_highscore` (drop the hardcoded `"classic"`).
- Optional tiny follow-on (NOT in scope): `MiniMetroEnv` map-symmetry (`env.py` accepts no `map_definition`; `PlayerPixelEnv` does). The pixel RL path already runs chosen maps, so GM-09f's deliverables don't need it. Flag, don't fold.

Rationale: the save-schema migration touches a byte-frozen fixture + strict exact-key validation + a version bump — it deserves its own focused TDD + dual review, not to be entangled with UI rect-layout and a separate persistence schema. The plan review should confirm/refine this split.

## GM-09f (save-schema v2) design

Blessed by D-026 / README:230 — "a future schema version that introduces map selection synthesizes `classic` for older saves."

1. **Versioning** (`src/save_schema.py`): add `SAVE_SCHEMA_VERSION_V2 = 2`; `SUPPORTED_SAVE_SCHEMA_VERSIONS = {V1, V2}`; `SAVE_SCHEMA_VERSION = SAVE_SCHEMA_VERSION_V2` (new saves are v2). Forward (>=3) still rejected.
2. **Fields**: v2 adds two top-level keys `mapId` (str) + `mapDefinitionVersion` (int). Make `_TOP_LEVEL_KEYS` VERSION-AWARE: `_TOP_LEVEL_KEYS_V1` (current set, no map) and `_TOP_LEVEL_KEYS_V2 = _TOP_LEVEL_KEYS_V1 | {mapId, mapDefinitionVersion}`; `validate_save` selects by the document's `schemaVersion` and runs `_exact_keys` against the matching set (so a v1 doc with map keys, or a v2 doc missing them, both fail closed).
3. **Serialize** (`src/save_game.py`): DELETE `_require_classic_map` (and its call). Emit `schemaVersion = V2` and `mapId`/`mapDefinitionVersion` from `mediator.map_definition` (default CLASSIC → `classic`/1). Validate the map is resolvable (`resolve_map` round-trips) before writing — a fail-closed guard replacing the old one.
4. **Deserialize** (`src/save_load.py:311`): read `mapId`/`mapDefinitionVersion` for v2; for v1 (keys absent) SYNTHESIZE `classic`/1. `resolve_map(map_id, version)` → pass `map_definition=` into `Mediator(...)`. A v1 save thus loads as CLASSIC exactly as today.
5. **`save-v1.json` (the byte-frozen fixture)**: it stays BYTE-IDENTICAL on disk (a historical v1 document) and still LOADS (v1 path synthesizes classic@1) — its SHA `d34736a6…`/length-15442 pins are UNCHANGED. BUT `serialize_game` now emits v2, so the load→re-save idempotence test (`test_frozen_fixture_matches_the_freeze_recipe_and_loads`: `canonical_save_bytes(serialize_game(loaded)) == v1_payload`) BREAKS — a v1 load re-saves as v2 (an upgrade, different bytes). Handle by: (a) keep the v1 frozen-bytes + loads-successfully assertions; (b) change the idempotence assertion to "a v1 load re-saves as a VALID v2 document whose `mapId`==classic"; (c) add a NEW frozen `save-v2-classic.json` (a v2 CLASSIC save) and assert v2 round-trip idempotence (`canonical_save_bytes(serialize_game(load(v2))) == v2_payload`), plus per-map v2 round-trips for river/delta/lake.
6. **Flip the rejection test**: `test_gm09a_maps.py::test_serialize_rejects_a_non_classic_map` asserts a river map RAISES on serialize → rewrite to assert it SERIALIZES + round-trips (map identity preserved).

## PLAN v2 (revised after dual plan review — both lanes REVISE, direction + split confirmed sound)

Harness: 2 BLOCKER + 4 MAJOR. Codex: 6 MAJOR (corroborated the guard). Both confirmed: the split is right, the save unit is verifiable via the programmatic `Mediator(map_definition=...)` path (no menu/env change needed), v1-load stays byte-identical (`Mediator(map_definition=CLASSIC)` ≡ `Mediator()` — same singleton; deserialize overwrites RNG + discards the construction pool), and the blast radius is contained (RL manifest / recursive-checkpoint / high-scores are separate schemas; `SAVE_SCHEMA_VERSION` has one non-test consumer). Folded:

1. **STRUCTURAL serialize guard (both BLOCKER/MAJOR — the load-bearing fix).** v2 persists only `(mapId, version)` and rebuilds terrain from the registry on load, so "resolvable" is NOT enough — a forged/drifted classic (registered id, extra terrain) would serialize as `classic@1` and reload as the real terrain-free CLASSIC (empirically: 2 of 20 seed-0 CLASSIC stations sit in RIVER's band). Guard: reject unless `mediator.map_definition == resolve_map(map_def.map_id, map_def.map_definition_version)` (frozen-dataclass structural `==`), generalizing GM-09b's `== CLASSIC`. `resolve_map` raises first on unknown id / unsupported version (fail-closed). Keep `test_gm09b_river::test_forged_classic_with_terrain_is_rejected` GREEN; flip only the REGISTERED-non-classic rejections using CANONICAL RIVER/DELTA/LAKE for the positive tests, and ADD forged-river/delta/lake rejection cases.
2. **STATE-legality validation on serialize AND load (Codex MAJOR).** A structurally-valid identity paired with an illegal state is still corrupt: relabeling a CLASSIC save `river@1` strands stations in the water, and a swapped state can have `consumed_tunnels > num_tunnels` (both verified). Add a shared `_require_legal_map_state(mediator)`: for a map with `spawn_regions`, every station centre lies in some region; if `num_tunnels is not None`, `consumed_tunnels <= num_tunnels`. Call on serialize (reject saving an illegal state) and after reconstruction on load (reject a forged save). Tunnel counts stay DERIVED (no persisted counter). TDD: save a RIVER game with a committed river-crossing line, load, assert `consumed_tunnels`/`available_tunnels` match; and an exhausted-budget round-trip.
3. **`validate_save` TWO-PHASE ordering (both MAJOR).** Move version handling BEFORE key selection, fail-closed with named `ValueError` (never KeyError/TypeError): (1) `type is dict`; (2) safe-value; (3) read `schemaVersion` defensively (`.get`; missing/non-exact-int/bool → named ValueError); (4) support-check (`in SUPPORTED`, else forward-rejected); (5) select `_TOP_LEVEL_KEYS_V1`/`_V2`; (6) `_exact_keys`; (7) header/rest. Negative corpus: missing/bool/float/list/null/0/3 schemaVersion, v1-doc-with-map-keys, v2-doc-without-map-keys — all clean rejections.
4. **mapId/mapDefinitionVersion validators (harness MAJOR).** Mirror `rl/manifest_schema._validate_map_identity`: `mapId` non-empty ASCII string, `mapDefinitionVersion` positive non-bool int. So a v2 doc with `mapId:123` fails `validate_save` (not just "won't load"). Unknown-but-well-typed map id: defer to `resolve_map` at load (fail-closed there) — stated choice.
5. **`save-v2-classic.json` = the EXACT canonical transform of frozen v1 (Codex MAJOR).** Because a v1→v2 upgrade only changes the header, it is DETERMINISTIC: create it by `canonical_save_bytes(serialize_game(load_game(save-v1.json)))` — identical to v1 except `schemaVersion:2` + sorted-inserted `mapId:"classic"`/`mapDefinitionVersion:1`, retaining `pauseReasons:["menu"]`. Tests: (a) v1 load→re-save == save-v2-classic.json byte-for-byte (the upgrade is pinned); (b) v2 load→re-save idempotent (== itself); (c) per-map v2 round-trip (river/delta/lake serialize→load→`map_definition` equal); (d) both hash-seed workers produce the v2-classic bytes.
