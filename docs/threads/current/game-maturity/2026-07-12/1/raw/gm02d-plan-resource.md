# Initial refutation

1. HIGH — default promotion is underspecified and can silently select the wrong layout. `src/rl/policy.py:16` has only `DEFAULT_FRAME_STACK=8`; `build_vector_env()` at `src/rl/training.py:164-166` derives a *contiguous* descriptor when history is omitted. Setting that integer to 12 would produce contiguous-12, not `decision-history-v1`; changing only train CLI would split fresh defaults across callers. Correct plan: introduce one named default-history factory/constant in `rl.history`, make both fresh CLI resolution and `build_vector_env(history=None, frame_stack=None)` use it, retain `--frame-stack N` as explicit contiguous ablation, and add tests for fresh CLI -> env -> manifest exact descriptor/fingerprint plus resume preserving persisted history. Do not change `DEFAULT_FRAME_STACK` alone.

2. MEDIUM — the “182.25 MiB float32 recurrent batch-64 tensor” is only the *nominal unpadded* 12-frame tensor. Contrib `RecurrentRolloutBuffer._get_samples()` sequence-pads every nominal 64-index batch; even with no episode reset, a batch straddling env boundaries can have nearly 2x padded rows. Keep 182.25 MiB labeled nominal, but capture every yielded minibatch’s observation shape/dtype/bytes, valid `mask.sum()`, padded rows, maximum/total padded rows, and normalized CNN-input float bytes via a no-copy forward pre-hook. Rename `4096 / train_seconds` to “valid transition-row rate”; actual optimizer compute rows are the summed padded rows and should be reported separately.

3. MEDIUM — clean collection/train timing is feasible but must be specified. `RecurrentPPO.learn()` exposes rollout callbacks but no train callback. Wrap the live model instance’s `collect_rollouts()` and `train()` (or use a benchmark-only subclass) around one `learn(total_timesteps=2048)` call: call 1 warm-up, call 2 measured. Time with `perf_counter_ns`; assert exactly two successful 128-step collections and two train calls, exactly 1024 transitions/call, 4 epochs, batch 64, and no early stop. A unit fake should prove setup/reset time is excluded and warm-up cannot leak into measured counters.

4. MEDIUM — preregister memory-window/completeness semantics. Gate on maximum aggregate current working set over the *entire candidate lifecycle* (setup + warm-up + measured + teardown), and optionally report phase peaks; otherwise setup/worker allocation failures can be omitted. Toolhelp recursion must fail a repeat if an enumerated root/descendant cannot be opened or sampled, record actual max inter-sample gap (not just requested 100 ms), and test root+child+grandchild discovery/PID exit. Call this “aggregate process-tree working set,” not unique RAM/RSS (shared pages are double-counted).

5. MEDIUM — fallback promotion gate is ambiguous. Text says only twelve may promote, then mentions ten fallback without explicitly granting it the same gates. State: 12 is tested first; if it fails, 10 may promote only against the same all-repeats/batch64/<=1.25x/<historical-cap/>=0.75x rules, with fresh 8-contiguous controls interleaved in the fallback campaign. Otherwise GM-02d remains open. Use a synthetic-results gate test for equality boundaries, one failed repeat, batch drift, missing result, and fallback.

6. MEDIUM — “MAC estimate” needs scope/method. Define forward inference MACs per one valid observation row (shared CNN once, separate actor/critic LSTMs, action/value heads), report assumptions, and do not call it training MACs or multiply by 4096 without padded-sequence/backward accounting. Add known Conv/Linear/LSTM formula tests or omit MACs rather than emit an unverifiable number.

7. LOW — warm-up does not guarantee offset 128 is populated after an episode reset. Assert/record no done per slot during the 128 warm-up steps (or expose/record per-slot valid age); otherwise zero-filled anchor use differs across candidates.

8. LOW — the 3.909 GiB historical cap is rounded and its old sampler/raw bytes are absent. Keep it as a conservative historical guardrail, define exact comparison units/bytes in the gate artifact, and do not claim method-equivalence. The fresh relative baseline is the defensible primary comparison.

Math/storage claims otherwise check out: fast single frame 62,208 bytes; 12 rollout 729 MiB; 12 one-step 8-slot output 5.6953125 MiB; 129-frame 8-slot ring 61.224609375 MiB; 8-multiscale rollout 486 MiB with the same 61.224609375 MiB ring. Baseline 8-contiguous ring is only 3.796875 MiB. Machine feasibility looks adequate for the run (63.724 GiB physical, 41.677 GiB available at check, 32 logical CPUs), but promotion still needs measured peaks because padded float tensors/activations dominate transient memory.
