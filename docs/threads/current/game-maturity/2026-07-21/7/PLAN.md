## Canonical carriage and capacity model

- Add config-owned `num_carriages = 2`, `carriage_capacity = 6`, carriage geometry/colors/gap, carriage control layout/style, and a fourth-HUD-line exclusion rectangle. These are provisional playable defaults, not a GM-11 balance conclusion.
- Add an attached `Carriage` entity with a nonempty unique ID, immutable positive integer capacity, fixed render shape, and no passenger collection. Unassigned units remain fungible and have no live assignable object pool. A Carriage identity retained only through a successfully detached historical Metro/Path object graph is retired: it is never counted, checkpointed, or reattached, and a later assignment always creates a fresh identity.
- Add one canonical ordered `Metro.carriages: list[Carriage]`. There is no `Mediator.carriages` pool. Every attached identity and ID occurs exactly once across canonical global Metros; carriage-list objects are real distinct lists and cannot be shared or rebound during a successful transaction.
- Add writable `Mediator.num_carriages`, read-only `assigned_carriages`, and read-only `available_carriages = max(0, num_carriages - assigned_carriages)`, where assigned counts only carriages attached to the canonical global `Mediator.metros` order. Within-cap state additionally satisfies `available + assigned == total`; direct over-cap compatibility clamps availability at zero and never deletes attachments.
- `Metro` remains the sole passenger `Holder`. It stores writable compatibility state `_base_capacity`; `capacity` becomes a derived property equal to base capacity plus its ordered carriage capacities. Assigning `metro.capacity = value` accepts only an integer-not-boolean total at least `max(attached_capacity_sum, len(passengers))`, then atomically updates base capacity so the getter immediately equals that total; invalid, boolean, negative, below-attached, or passenger-overfilling writes raise without mutation. Attachment and detachment never incrementally edit the base value. Canonical runtime state requires nonnegative integer-not-boolean base/total values, strictly positive integer-not-boolean Carriage capacities, and `len(passengers) <= capacity`.
- Default construction and assignment produce a six-seat locomotive with an empty carriage list. Reset restores two available carriages by constructing a fresh default game state; no old entity survives reset.

- Detach selects capacity-safe nonqueued candidates that have at least one carriage and whose remaining derived capacity is at least their current passenger count. It chooses the most-carriaged candidate, breaking ties by latest `path.metros` order, and removes that Metro's last carriage. The same full snapshot/postcondition and exception rules apply. The removed identity becomes retired historical state if an outside reference retains it; no live pool retains it and later attachment creates a new identity.

Exactly one selector is required; IDs are nonempty and unique, indices are real in-range integers and never booleans. To preserve the existing live fleet API, unrelated extra keys are ignored after selector validation; persisted v5 carriage actions remain exact-key-only. Unknown, ambiguous, duplicate-ID, detached, creating, malformed, terminal, queued-only, capacity-unsafe, exhausted, or otherwise inapplicable targets reject without time, seeded RNG, collection, pointer, observation, or checkpoint effects.
- Paused actions may attach or safely detach immediately; terminal actions reject. Attaching to an occupied Metro is valid. Capacity-safe occupied detachment is valid because it strands no rider. Queued locomotives cannot gain or lose carriages.
- Existing empty-locomotive unassignment carries its whole attached consist through the queue and refunds every attached unit only when exact locomotive detachment removes the owner from the canonical global collection. Immediate and delayed return share carriage validation and preserve exact composition on removal rollback; successful removal retires any externally retained composition identity. A late line-removal failure after global-Metro removal can leave an active Path with noncanonical retained Metros under the existing nontransactional lifecycle; GM-06c records that state as malformed, rejects carriage mutation/checkpoint generation, and defers reorder/rollback and truthfully restored availability to GM-06d rather than claiming partial-order safety.
