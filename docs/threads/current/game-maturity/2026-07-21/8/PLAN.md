## Case 1 - Occupied-locomotive queued return

`FleetManagement.unassignment_candidate` currently requires `not metro.passengers`, so a path whose locomotives all carry riders cannot return one. GM-06d widens selection: prefer the last empty nonqueued Metro exactly as today; only when no empty candidate exists, select the occupied nonqueued Metro with the fewest passengers, breaking ties by latest `path.metros` order. Both reviews confirmed the existing machinery then drains it correctly: boarding stays queue-blocked, unloads stay un-gated, `should_stop_at_next_station` forces every real-station stop, and `settle` detaches the emptied Metro from both collections the same tick it empties, refunding its whole consist through the derived counts exactly once.

- Guaranteed drain (review m1/F6; pacing per re-review NEW-3): when a queued Metro is stopped at a real station, ordinary destination/transfer service runs first through the unchanged oracle and cache; only when the pure oracle resolves no executable action while riders remain does the drain rule fire, force-alighting every rider matching the three sub-cases — no plan, a plan with no alight on this line, or a transfer blocked only by station capacity — in exact holder order as one atomic batch within that tick, under the D-024 overflow-permitted placement, with each alighted rider's `travel_plans` entry cleared and `wait_ms` reset before the ordinary re-plan sweep (re-review N5). The batch never touches `_station_service_action`, which stays `None` under the existing queued stop-override, so the checkpoint-v4 cache-equals-pure-oracle contract is untouched; departure and same-tick settle then proceed normally. A queued occupied return therefore completes within bounded travel and a deterministic replay trace; the full-transfer-station livelock is eliminated.
- `queue()` keeps its immediate-detach fast path gated on empty-at-station. For an occupied Metro it now preserves the service cache, `stop_time_remaining_ms`, and `boarding_progress_ms` (review m2) so an in-progress unload continues seamlessly; only the empty fast path clears service state as today.
- The widened selection extends the existing snapshot/rollback and canonical-state guards unchanged; occupied selection introduces no new settlement rule (review C3/C4 confirmed).

## Case 2 - Queue cancellation

Add public `Mediator.can_cancel_unassignment(path)` / `cancel_unassignment(path)` facades backed by `FleetManagement`: select the earliest queued Metro in `path.metros` order on the exact active path of a non-terminal game and atomically restore `is_unassignment_queued = False` with snapshot/rollback on any failure. Cancellation restores boarding permission, candidate production, and ordinary stop planning; service timing re-derives at the next per-tick reconcile seam, which zeroes fractional progress only when the resolved action identity changes (review C5 confirmed; the original "identical to never having queued" wording is deliberately softened per review m2). Rejection cases — terminal game, non-canonical queue state, inactive path, no queued Metro — have no time, RNG, collection, pointer, observation, or checkpoint effect. A live structured action `{"type": "cancel_unassignment", "path_id"|"path_index": ...}` follows the existing fleet selector/validation rules with unrelated-extra-key tolerance. Persisted gating (re-review NEW-2): `cancel_unassignment` is live/structured-only in GM-06d — it joins the known version-gated persisted-action sets in `recursive_contract.py` and agent preflight as explicitly rejected at every persisted version v1-v5 before environment construction or mutation, closing the unknown-type fall-through that would otherwise let a crafted legacy document execute the new mutation; a future contract version owns persisting it. No persisted schema changes: a cancelled queue leaves state a fresh recording could always have produced, and genuine v1-v5 matrices are unaffected.

## Case 3 - Transactional line removal with rider conservation

This case now leads the redesign (reviews B1/F1/M2/F2): `PathLifecycle.remove_path` becomes a snapshot/rollback transaction, and rider handling becomes conservation-first under D-024.

- Snapshot before any mutation: the exact touched footprint — button binding and `path_to_button`, `host.paths` order, `host.metros` order, every affected Metro's passenger list/service fields/queue flag/carriage list, every affected rider's holder membership, `travel_plans` entry, `wait_ms`, and `is_at_destination`, the global `host.passengers` membership, station passenger lists, `path_colors`/`path_to_color`, the complete delivery/progression cluster the credited alight can touch (re-review NEW-1: `deliveries`, `line_credits`, `unlocked_num_paths`/`unlocked_num_stations` in both aggregate and facade views, the `host.stations` binding, and station/button unlock-blink and lock state), and both seeded RNG streams across the whole transaction — with exact-identity restoration on any ordinary `Exception` and rethrow of non-`Exception` failures. A failed removal is a complete no-op even when an earlier alight already credited a milestone-crossing delivery; the half-removed-line shape the reviews proved unrepairable can no longer exist.
- Ordering (review F2/M1): per Metro, riders alight before any collection mutation; only then is the Metro removed from `host.metros` and its consist thereby refunded. Button detachment, plan invalidation, color release, path removal, and button reassignment follow only after every rider is safe.
- Alight placement (review F3): a Metro on a real station alights there. A mid-segment Metro alights at the station endpoint of its current `PathSegment` nearest its progress; on a `PaddingSegment` (which has no station endpoints) it uses the nearest adjacent `PathSegment`'s station in travel order. Placement is deterministic and covered for both directions, loops, and bends.
- Rider disposition (reviews M4/m3/F6): a rider whose alight station matches its destination shape is recorded as a genuine delivery through the existing delivery hook, not silently removed. Every other rider joins the station's passenger list under the D-024 overflow-permitted placement, has its `travel_plans` entry cleared and `wait_ms` reset to zero, and is re-planned by the existing `find_travel_plan_for_passengers` sweep; riders with no reachable destination remain ordinary waiting passengers under the normal overdue rules. No rider is ever deleted.
- Contract flips (review M5; third flip found during red authoring): `test_remove_path_cleans_passengers` in `test/test_mediator_paths.py` and the onboard-rider deletion assertions in `test/test_path_lifecycle.py` assert the old destructive behavior, and `test_rider_injected_after_queue_blocks_destructive_settlement` in `test/test_gm06b_fleet_queue.py` asserts a planless rider aboard a queued metro stays aboard at a real-station tick — under the Case 1 drain rule that exact scenario now force-alights the rider and settles the metro. All three are consciously rewritten to the conserving contract; this public `Mediator` behavior change routes through the high-risk adversarial review lanes. Implementation surfaced five further old-contract tests that are mutually unsatisfiable with the transactional conserving removal and are flipped under the same mandate: three `test_gm06a_locomotive_inventory` removal-failure tests (expecting propagated ordinary exceptions with persistent partial/malformed state — now expecting silent exact restoration), the self-scoped `test_late_line_removal_failure_is_explicitly_malformed_until_gm06d` in `test_gm06c_carriage_lifecycle`, and the rider-deletion/event-order assertions of `test_removal_order_snapshots_and_detached_graph_are_preserved` in `test_mediator_path_contract`; each flip preserves its still-valid assertions (detached-graph retirement, refund accounting translated to the transactional contract, non-`Exception` rethrow).

## Case 4 - Narrow reconcile for residual malformed fleet state

With transactional removal, the late-partial line-removal shape is eliminated at the source rather than repaired after the fact. What remains is a deliberately narrow, idempotent `FleetManagement.reconcile(host)` for residual non-canonical fleet shapes that legacy states or external mutation could still present (reviews M1/M3/F5): it clears `is_unassignment_queued` on a Metro absent from canonical `host.metros`, and drops a `path.metros` entry absent from `host.metros` only when that Metro's passenger list is empty. It refuses everything else — a non-empty globally absent Metro is never dropped (that would retire riders); its visibility mechanism (re-review N4) is the existing canonical-gate refusal: every fleet query/action keeps returning `False`/`0` while the Metro, its riders, and its composition remain present in `path.metros`, rendering, and structured observations, so the discrepancy stays observable instead of silently ratified. The stale-service-action repair from the draft is removed as redundant with the per-tick reconciliation. Reconcile runs in `mediator.increment_time` immediately before `settle` for same-tick recovery (review F4) and runs unconditionally — including paused and terminal states — because it is a no-op on canonical state and GM-07 must never serialize a repairable shape (review F7). Before/after assertions prove rider-count and resource-count conservation across every reconcile outcome.

## Acceptance

- A locomotive carrying riders can be returned: riders drain through ordinary unloads plus the D-024 forced-alight rule, the return completes within bounded travel, and the emptied locomotive settles and refunds its whole consist exactly once. No rider is deleted or frozen aboard.
- A queued return can be cancelled, restoring boarding/candidacy with riders, composition, and inventory intact; rejected cancellations have zero effects.
- Removing a line never deletes a rider and never leaks or duplicates a locomotive or carriage: destination-matching riders are credited deliveries, everyone else alights (overflow-permitted), re-plans or waits, and a failed removal restores exact prior state.
- Ordinary spawn/boarding/transfer capacity gates are byte-for-byte unchanged; an over-capacity station only drains.
- Residual non-canonical fleet shapes are either safely repaired by the narrow reconcile or left visibly rejected; reconcile never fabricates, moves, or retires a rider or resource.
- No persisted checkpoint/recursive/agent version, low-level action protocol, task fingerprint, or frozen artifact changes; genuine v1-v5 replay and agent behavior stay exact.
