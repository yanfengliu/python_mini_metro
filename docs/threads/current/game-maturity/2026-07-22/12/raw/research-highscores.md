# GM-07d research — high scores integration surface (synthesis of the verbatim lane report)

- Ranking metric: `progression.deliveries` (progression.py:25-26,96-100) is the monotonic lifetime-delivery objective; `record_path_purchase` subtracts only `line_credits` (progression.py:90-94), which is exactly why deliveries (not credits/score) is the roadmap's ranking value (D-002). The final score is `mediator.deliveries` at the instant `is_game_over` flips (passenger_flow.py:463); deliveries is already in `canonical_checkpoint` (recursive_checkpoint.py:414,430) so no schema change is needed to read it.
- Record triggers: the PLAYING->GAME_OVER promotion in `handle_event` (app_controller.py:98-100 — the same call that deletes the autosave) and, mirroring the D-027 delete, the window-close game-over race in main (main.py:150-154). These are mutually exclusive per game: after promotion the state is GAME_OVER (a later quit does not record), and the window-close quit exits the process — so gating both identically to the autosave delete records exactly once.
- Persistence to reuse: the save-local atomic writer `save_game` (save_game.py:251-269, mkstemp->fsync->os.replace, mkdir parents), `canonical_save_bytes` ASCII recipe (save_schema.py:240-251), the strict validator primitives from save_schema_records, and the dup-key-rejecting strict load (save_load.py:327-347). Directory `config.save_dir_name = "saves"` is already gitignored (`/saves/`). There is NO existing ranked/capped multi-entry schema — the list shape is new; only the scalar validator idioms and the writer transfer. D-026 explicitly rejected the RL `write_json_atomic` (UTF-8, fail-open) for player persistence.
- Rules identity: `SAVE_RULES_VERSION = "rules-v1"` (save_schema.py:41); no separate balance identity exists (D-026's same-commit-bump policy governs it). The key now is map="classic" (D-026's reserved placeholder) + rulesVersion, stored as separate fields for GM-09 extensibility to `map_id` + `map_definition_version` (roadmap PLAN.md:218).
- Isolation: the AST scan (test_gm07b_save_determinism.py:284-311) forbids save-module imports in env/agent/recursive/checkpoint/rl; a new `highscores.py` importing save_schema helpers must be added to that forbidden set and kept out of every headless surface. AppController is already persistence-agnostic and takes injected seams (app_controller.py:66-83; main.py:120-134) — a `highscores` seam wired the same way keeps the controller and headless play import-clean and never writes a file from `env`.
- Display: the game-over overlay is drawn by `game_renderer._draw_game_over` (game_renderer.py:429-494), which is at 494/500 lines — NEAR CEILING; adding an indicator there risks the hard limit and breaks the exact-text assertions in test_game_renderer.py:377-428. Drawing a compact "new best" indicator in `main` AFTER the renderer's game-over frame keeps game_renderer untouched.
- Budgets: app_controller 222, main 225, menu_screens 158, config 195, progression 100 (all room); game_renderer 494 (do not grow); tests test_gm07a_app_controller 481 and test_gm07b_save_schema 494 are at/near ceiling — add new test_gm07d_* modules.
- Test idioms: the GM-07c seam-injection template (test_gm07c_autosave_controller.py — spy SimpleNamespace seam, _RecordingMediator, require-signature guards); the GM-07b atomic-write/schema/fuzz/frozen-fixture idioms; test_game_renderer game-over overlay assertions.

Open decisions (resolved as D-028): file name/schema/cap; the map+rules key shape; where the record fires (both surfaces, gated); "new best" indicator vs list and its draw surface; standalone module vs main-folded; read-only-on-mediator. Risks: the game_renderer ceiling; the render-test lockstep; the isolation-scan extension; encoding divergence (ASCII fail-closed, not RL UTF-8); corrupt-file tolerance must START EMPTY (unlike a save's hard reject) so a bad leaderboard never blocks game-over; and the exactly-once record across the two game-over surfaces.
