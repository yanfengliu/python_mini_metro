## Design (D-028)

- New `src/highscores.py` (< 500): the leaderboard document type, strict validation, ranked insertion, and persistence — reusing the save infrastructure without importing gameplay. Constants `HIGHSCORES_SCHEMA_VERSION = 1`, `HIGHSCORES_STATE_CONTRACT = "mini-metro-highscores-v1"`, `HIGHSCORES_PER_KEY_CAP = 10`, and `HIGHSCORES_MAP_CLASSIC = "classic"`; it imports the scalar validator helpers (`_object`/`_array`/`_exact_keys`/`_nonnegative_int`/`_string`/`_int`) and `canonical_save_bytes` from the save modules — so it joins the persistence-isolation set (the `SAVE_RULES_VERSION` rules identity is owned by `main`, which passes it explicitly; the implementation review moved it there so `record_score` never defaults the key).
  - `validate_highscores(document) -> None`: exact top-level keys `{schemaVersion, stateContract, entries}`, `schemaVersion` an int in the supported set (forward versions rejected), `stateContract` the pinned literal, `entries` an array of exact-key `{map, rulesVersion, deliveries}` objects with string map/rulesVersion and nonnegative-int deliveries; duplicate JSON keys rejected via the shared `object_pairs_hook`. All rejections raise `ValueError`.
  - `load_highscores(path) -> Document`: read → strict-validate → return, but START EMPTY on any `Exception` (review MAJOR-2 — broader than `(ValueError, OSError)` because a deeply nested document raises `RecursionError`, which is neither) — a missing, unreadable, malformed, forward-version, or pathologically nested file yields the empty document and never raises. This is the deliberate difference from `load_game` (which fails closed): a cosmetic leaderboard must never block play.
  - `save_highscores(document, path)`: `canonical_save_bytes` through an atomic writer inlined in `highscores.py` (review MAJOR-4 — the GM-07b writer is inlined inside `save_game.save_game` coupled to `serialize_game` and is not importable, so `highscores.py` adds its own `os`/`tempfile` and repeats the reviewed mkstemp→fsync→`os.replace`→`finally unlink` shape rather than touching the reviewed save module); an interrupted write retains the previous valid leaderboard.
  - `record_score(document, *, deliveries, map, rules_version) -> RecordResult`: returns a NEW document (pure — never mutates the input) with the entry appended, then all entries stored in a canonical order — sorted by `map` ascending, then `rules_version` ascending, then `deliveries` descending, with exact ties broken by a stable sort over append order (so the earlier-recorded entry precedes) — and each `(map, rules_version)` group truncated to `HIGHSCORES_PER_KEY_CAP`. This makes the document fully deterministic given the record sequence (which tests control; cross-key order is fully determined by `map`/`rules_version`), reconciling review MINOR-8's reproducibility goal with MINOR-5's tie semantics — the two cannot both be literal since insertion order is itself history, and stable-over-append is the coherent choice. The `RecordResult` carries the new entry's 1-based `rank` within its key (`None` if it fell outside the cap) and `is_best`. Pinned rank semantics (review MINOR-5): a new entry equal to the current best ranks 2 with `is_best` false (it sorts after the equal earlier best), and the first entry on an empty board — even `deliveries == 0` — ranks 1 with `is_best` true. Ranking and the cap apply per `(map, rules_version)` key so other keys are untouched (a forward-version file that started empty is overwritten on the next save — an accepted consequence of the cosmetic stance, review MINOR-9).
- `AppController` gains one optional inert `highscores` seam beside `autosave` (a `record(deliveries) -> RecordResult | None` callback; `None`/absent → no recording, exactly as today — the regression guard). At the PLAYING→GAME_OVER promotion (right where `_autosave_delete` fires), the controller reads `mediator.deliveries` ONLY inside the `self._highscores is not None` guard (review MAJOR-3 — a seam-less controller must not read `deliveries` at all, since the regression-guard stubs have no such attribute) and stores the returned result in a public `last_highscore_result` attribute (renamed from `game_over_result` to avoid colliding with the unrelated `mediator.game_over_result` stub return — review MINOR-6). The promotion UNCONDITIONALLY (re)assigns `last_highscore_result` — to `None` when the seam is absent or returns `None` — so a restart on the reused controller can never show a stale best (review MINOR-7). The controller never mutates the mediator; a seam-less controller is behaviorally identical to pre-GM-07d.
- `src/main.py` supplies the `highscores` seam as a closure binding `Path(save_dir_name)/"highscores.json"` (a patchable module-level `HIGHSCORES_PATH` and `record_highscore(mediator)` wrapping load→`record_score`→save, swallowing `Exception` best-effort exactly as the proven `write_autosave` does — review MAJOR-2, so no game-over record ever crashes play), passes it to `AppController`, and applies it at the window-close game-over race with the same `is_game_over` + `state in (PLAYING, PAUSE_MENU)` gate as the autosave delete — recording exactly once (mutually exclusive with the promotion path). After the renderer draws the game-over frame, `main` draws a compact best indicator from `controller.last_highscore_result` via a small public `menu_screens` primitive (reusing the `draw_notice` idiom), gated on `state == GAME_OVER` and `is_best` — `game_renderer` is not touched, so its 494-line count and its exact-overlay tests are unchanged.
- Isolation: `env.py`, `agent_play.py`, `recursive_playtest.py`, `recursive_checkpoint.py`, and `rl/*` gain no `highscores`/save import; the persistence AST scan is extended to name `highscores` in its forbidden set. Recording fires only through the injected seam at the controller/main game-over surfaces, so headless and RL play never write a leaderboard.

## Acceptance

- Finishing a run records its lifetime `deliveries` to `saves/highscores.json` under the `(classic, rules-v1)` key, ranked descending and capped per key, written atomically so an interrupted write keeps the prior leaderboard; a new best shows a compact indicator on the game-over screen.
- The record fires exactly once per game across the game-over-screen and window-close-at-game-over surfaces; a corrupt, missing, or forward-version leaderboard starts empty and never crashes or blocks play; the recorder mutates no mediator state and changes no checkpoint or observation.
- Headless and programmatic play write no leaderboard (isolation scan green, extended to name the module); no schema, protocol, or frozen-artifact change; all budgets and gates hold and `game_renderer` is unchanged.
