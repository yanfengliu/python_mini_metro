## Design (D-029)

- New `src/settings.py` (< 500): a typed frozen `Settings`, strict validation, and versioned JSON persistence that fails safe to defaults. Constants `SETTINGS_SCHEMA_VERSION = 1`, `SETTINGS_STATE_CONTRACT = "mini-metro-settings-v1"`. `Settings` fields with defaults: `fullscreen: bool = False`; `master_volume: int = 100`, `music_volume: int = 100`, `sfx_volume: int = 100` (integer percents 0-100 so canonical bytes stay unambiguous); `reduced_motion: bool = False`. `DEFAULT_SETTINGS = Settings()`. It imports `canonical_save_bytes` and the scalar validators (`_object`/`_exact_keys`/`_bool`/`_int`) from the save modules and adds a local `_percent_int` (int in [0, 100], non-bool) — so it joins the persistence-isolation set — and imports no gameplay.
  - `validate_settings(document) -> None`: exact keys `{schemaVersion, stateContract, fullscreen, masterVolume, musicVolume, sfxVolume, reducedMotion}` checked before field access, `schemaVersion` an int in the supported set (forward versions rejected), `stateContract` the pinned literal, strict `bool` `fullscreen`/`reducedMotion` (int-as-bool rejected), `_percent_int` volumes; duplicate JSON keys rejected via the shared `object_pairs_hook`. All rejections raise `ValueError`.
  - `settings_to_document(settings) -> dict` and `settings_from_document(document) -> Settings`: the document<->value mapping (`settings_from_document` validates strictly, then constructs).
  - `load_settings(path) -> Settings`: read -> strict-validate -> construct, but FALL BACK to `DEFAULT_SETTINGS` on any `Exception` (including `RecursionError` from a pathologically nested file). A missing, unreadable, malformed, or forward-version file yields the typed defaults and never raises — the deliberate difference from `load_game`, matching the highscores cosmetic-file stance.
  - `save_settings(settings, path)`: validate the derived document, then write `canonical_save_bytes` through a save-local copy of the reviewed GM-07b mkstemp->fsync->os.replace->finally-unlink atomic writer; RAISES on failure so an interrupted write keeps the previous valid file.
- `AppController` gains `AppScreen.SETTINGS` and one optional inert `settings` seam (`load() -> Settings`, `save(Settings) -> None`). The controller holds a public `current_settings: Settings` — loaded via `seam.load()` at construction, or `DEFAULT_SETTINGS` when the seam is absent — and a private `_settings_origin: AppScreen` so `back` returns to whichever screen opened SETTINGS. `SETTINGS` is reachable from `TITLE` and `PAUSE_MENU` via a `settings` entry APPENDED after the existing entries in `title_layout`/`pause_menu_layout`, preserving existing button rects (the plan review confirmed the existing menu tests are property-based, not golden, so none needs editing — dropping this plan's earlier "menu byte tests must be edited" premise). Both handlers need an explicit branch: `_handle_title` keys off explicit layout keys so it gains an `elif` for `settings`; `_handle_pause_menu` arms then dispatches on the armed key so it gains an `elif armed == "settings"`. Entering SETTINGS from PAUSE_MENU must NOT release the `menu` pause reason (it records `_settings_origin`, disarms the armed control, and keeps the hold), and `back` returns to `PAUSE_MENU` with the hold intact; from TITLE, `back` returns to `TITLE`. Each SETTINGS control (fullscreen toggle, three volume steppers, reduced-motion toggle) builds a NEW `Settings`, updates the in-memory `current_settings` (so the live fullscreen/reduced-motion consumers work), and calls `seam.save(new)` only when the seam is present. A seam-less controller starts at `DEFAULT_SETTINGS`, updates `current_settings` in memory on an edit but writes no file (the regression guard asserts no persistence call and the pre-GM-08a screens are unchanged); the controller never touches the `Mediator`.
- `src/main.py` also gains the settings render branch: `run_game`'s draw block gains an `elif state == AppScreen.SETTINGS: draw_settings_menu(game_surface, controller.current_settings)` so the screen paints its own chrome rather than a bare game frame.
- `src/main.py` supplies the seam as a closure over a patchable module-level `SETTINGS_PATH = Path(save_dir_name) / "settings.json"` (`load_settings`/`save_settings` wrappers that swallow `Exception` best-effort exactly as `write_autosave`/`record_highscore` do), passes it to `AppController`, and applies the two live consumers each frame: (1) `fullscreen` — the STARTUP `set_mode` stays exactly `((screen_width, screen_height), pygame.RESIZABLE)` (preserving the `test_main.py` set_mode-once contract), and a separate `applied_fullscreen` local (initialized `False`) drives a LATER `set_mode` only when `controller.current_settings.fullscreen != applied_fullscreen`: windowed flags stay `pygame.RESIZABLE`, fullscreen flags are `pygame.FULLSCREEN | pygame.SCALED`, and the return value is REASSIGNED to `window_surface` so `get_window_size`/blit read the new mode; the virtual game surface and letterbox viewport already handle arbitrary window sizes and are unchanged. (2) `reduced_motion` — pass `controller.current_settings.reduced_motion` into `renderer.draw(...)`. Volumes are stored but have no consumer until GM-08b (the SETTINGS screen labels them as such). The existing `test_main.py` set_mode-once assertion (`assert_called_once_with((screen_width, screen_height), pygame.RESIZABLE)`) is a REQUIRED contract edit: it must patch `main.SETTINGS_PATH` to a temp/absent path so the construction-time load resolves to `DEFAULT_SETTINGS(fullscreen=False)` and the assertion cannot couple to a developer machine's persisted `fullscreen=True`.
- `src/ui/menu_screens.py` gains a byte-stable `draw_settings_menu(surface, settings, ...)` (reusing the `_stacked_buttons`/`_draw_button`/`_draw_heading` idioms) that paints the current values, plus a `settings` entry in `title_layout` and `pause_menu_layout` and a `settings_menu_layout` for the control hitboxes.
- Reduced-motion gating (revised per plan review — the blink phase is NOT in `game_renderer`; it lives in the entity/ui leaves, so there is nothing to extract there): the on-off predicates are `station.is_unlock_blink_visible` (`station.py`), `passenger.is_warning_blink_visible` (`passenger.py`), and the path-button unlock blink (`path_button.py`); the station snap blip is a one-shot expanding ring. `reduced_motion` reaches those leaves through the per-frame render-resources object already threaded to entity draws: `GameRenderer.draw(surface, state, alpha, reduced_motion=False)` stashes `self.resources.reduced_motion` once at the top (about +2 lines, so `game_renderer` stays under 500 with no extraction), and `station.draw`/`path_button.draw` — which already receive `resources` via `_call_flexibly(..., resources=self.resources)` — read it. The passenger-warning blink is reached only through `Holder.draw`/`station.draw` and `Metro.draw`, which do not pass `resources` to `passenger.draw` today, so `reduced_motion` is forwarded to `passenger.draw` as an explicit `reduced_motion=False` param through those two call chains. Semantics: under reduced motion the passenger-warning and station/path-button unlock blinks are held VISIBLE and steady (the state is still conveyed, just not flashing) and the one-shot snap blip is SUPPRESSED (pure animation, no information). HARD INVARIANT: `reduced_motion` defaults `False` and every `False` branch is byte-for-byte identical to today's output, so the `game_renderer` byte/surface tests and the `renderer.draw` reproducibility tests stay green; a red test pins each blink's steady-on and the blip's suppression under `True`. If the explicit passenger forwarding still pushes `game_renderer` past 500, an existing cohesive block (`_draw_redraw_preview` or `_draw_game_over`) is extracted — never a nonexistent blink helper.
- Isolation: `env.py`, `agent_play.py`, `recursive_playtest.py`, `recursive_checkpoint.py`, and `rl/*` gain no `settings` import; the persistence AST scan is extended to name `settings` in its forbidden set. The store fires only through the injected seam at the controller/main surfaces, so headless and RL play never construct or read a settings file.

- The `SETTINGS` screen is reachable from the title and pause menus, edits the fullscreen toggle, the three volumes, and reduced-motion, and persists each change atomically; fullscreen applies to the live window and reduced-motion holds the passenger-warning and unlock blinks steady while suppressing the one-shot snap-blip rings.
- Headless, agent, and RL play construct and read no settings file (isolation scan green, extended to name the module); settings touch no `Mediator` or `config` balance and change no save schema, checkpoint, observation, or frozen artifact; all budgets and gates hold, every touched renderer/entity file stays under 500 lines, and the renderer produces byte-identical output when `reduced_motion` is `False` (its default), so no existing render or menu test regresses.
