## 1. Fullscreen mid-loop `set_mode` — MAJOR (two concrete gaps)

`main.run_game` captures `window_surface` once at `main.py:134` (`window_surface = pygame.display.set_mode((screen_width, screen_height), flags)`, `flags = pygame.RESIZABLE` at :133) and reuses that local every frame for sizing (`get_window_size(window_surface)` :185), fill (:248), and blit (:263). `pygame.display.set_mode` returns a *new* display surface.

- **Gap 1a (reassignment/flags):** the plan never states the return value must be REASSIGNED to `window_surface`, that an `applied_fullscreen` local must gate the call to fire only on change (calling set_mode every frame recreates the window), and that flags matter — the robust choice is `pygame.FULLSCREEN | pygame.SCALED`. The virtual `game_surface` + letterbox viewport (main.py:186-263) DO survive a window-mode change (that plan claim is sound) but only if `window_surface` is reassigned so `get_window_size` reads the new mode.
- **Gap 1b (existing hard-pinned test):** `test/test_main.py:149-151` asserts `set_mode.assert_called_once_with((screen_width, screen_height), RESIZABLE)` in a `run_game(max_frames=1)` PLAYING run with the real `AppController`. The initial set_mode must stay exactly `((sw,sh), RESIZABLE)` and fullscreen application must be a separate later set_mode. `AppController.__init__` will call the seam's `load()` (D-029), resolving the real `SETTINGS_PATH`; on CI (no file) → DEFAULT(fullscreen=False) → set_mode once → green, but on a dev machine with persisted `fullscreen=True` → set_mode twice → assertion FAILS, coupling the local gate to user state. Amendment: name `test_main.py` in the affected list; make `main.SETTINGS_PATH` patchable and patch it in that test. The loop-harness tests (`test_gm07a/c/d_run_game_loop.py`) set `set_mode.return_value = window` permissively and assert only advance/draw/dispatch — they stay green. FULLSCREEN under SDL dummy belongs in the windowed impl-review lane.

## 5. Persistence correctness — SOUND; one over-caution

Int-percent [0,100] over float [0,1] is correct for canonical bytes (allow_nan=False, sort_keys=True; floats invite repr ambiguity). Validator names exist and are exported: `_object` (save_schema_records.py:49), `_exact_keys` (:115), `_bool` (:67), `_int` (:73); `canonical_save_bytes` (save_schema.py:240), `SAVE_RULES_VERSION` (:41) — highscores.py:22-30 already imports this set. Strictness sufficient: `_bool` is `type(value) is not bool` → rejects int-as-bool; `_int` is `type(value) is not int` and `type(True) is bool` → `_int` rejects True. So `_percent_int = _int + range[0,100]` rejects a True volume, and strict-bool for fullscreen/reducedMotion is satisfied by `_bool` directly — no dedicated strict-bool check needed (plan over-caution, harmless). Isolation scan: `SAVE_MODULE_NAMES` at test_gm07b_save_determinism.py:29-35 (contains highscores); GM-07d mirror `TestGM07dPersistenceIsolation` at test_gm07d_menu_screens.py:104-160. Add "settings" + a test_gm08a isolation test in the same shape; baseline stays green (settings.py imported only by main.py/app_controller.py, neither in the scan target set; AST scan is direct-import only).
