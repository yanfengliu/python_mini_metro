## Screen states and AppController

- New `src/app_controller.py` (< 500 lines) owns an explicit screen-state machine for the human entry path only: `TITLE`, `PLAYING`, `PAUSE_MENU`, and `GAME_OVER`. A `SETTINGS` state is deliberately deferred to GM-08a, which supplies the typed settings store it would edit; the enum is extensible and nothing references a settings screen until it exists. Because this amends the GM-07 roadmap scope bullet ("title, pause, settings, and game-over navigation states"), the deferral is recorded as an explicit decision line in the parent `DECISIONS.md` within this transaction (review F6), not left as a per-unit aside.
- `run_game` keeps ownership of the window, surfaces, clock, viewport, and pygame pump, but delegates per-event and per-frame decisions to the controller: which events reach `session.dispatch`, which trigger navigation, whether the simulation advances this frame, and which screen paints above the gameplay frame. The controller never touches `pygame.display`; it consumes already-converted virtual-coordinate events plus raw keyboard events, so it is fully testable headlessly.
- `TITLE` offers New Game and Exit (keyboard and mouse). The game triple is constructed eagerly: a `Mediator`/`GameRenderer`/`GameSession` exists while sitting on `TITLE` (review F5), so the state machine is a pure presentation/dispatch selector and New Game simply replaces the triple through the shared reconstruction path (fresh random seed, no `Mediator.reset()`), the same path used by restart-from-pause-menu and restart-from-game-over (research R8).
- Construction seam (review F2): `main.run_game` supplies the construction callable — a closure over the `main` module's own `Mediator`/`GameRenderer`/`GameSession` globals, preserving every existing `patch("main.…")` and source-module patch seam in `test/test_main.py` — and the controller owns only *when* to invoke it. The loop reads the current mediator/renderer/session through the controller every frame, so a menu- or title-restart rebinding is observed immediately; a red test proves the frame after a menu-restart dispatches and draws through the new session and renderer.
- Start-state selection (review F4): `run_game` auto-selects `PLAYING` whenever `max_frames is not None` — which covers all four existing direct-call tests and the `PYTHON_MINI_METRO_MAX_FRAMES` runpy hook with zero call-site edits — and starts at `TITLE` otherwise; an explicit `start_state` keyword overrides both for new tests.
- `PLAYING` behaves exactly as the current loop's playing branch: viewport-mapped mouse events and converted events to `session.dispatch`, simulation advanced by `session.advance(elapsed_ms)`.
- `PAUSE_MENU` opens only through the new Escape binding while playing (verified free: only the game-over branch binds Escape today, and gesture cancellation is mouse-based); Space remains the gameplay pause toggle untouched. On the PLAYING→PAUSE_MENU transition the controller first dispatches the existing letterbox cancel — `MouseEvent(MOUSE_UP, Point(-1, -1))`, whose no-release-actions cancel semantics are already pinned by the loop tests — so a mid-drag redraw/creation/handle gesture is cleanly abandoned before the `menu` reason is held, leaving no armed gesture or half-drawn preview behind the menu (review F3). The menu holds the `menu` reason for its whole lifetime, renders Resume / Restart / Exit-to-title above the frozen gameplay frame, and closes by releasing only the `menu` reason. While the menu is open, gameplay input is not dispatched; Space therefore cannot dismiss or bypass the modal state (D-010), because the user-reason toggle is unreachable and the menu reason is owned solely by the controller. A red test opens the menu mid-drag, releases over the menu, resumes, and asserts no stale gesture and zero release actions.
- `GAME_OVER` absorbs the loop-inline branch (research R7): the controller routes R/Escape/click through the existing `handle_game_over_click` rects and actions with identical outcomes (restart reconstruction; exit). The game-over overlay keeps rendering inside `GameRenderer` so gameplay and game-over bytes stay identical; title and pause-menu chrome render in a new `src/ui/menu_screens.py` (< 500) invoked from the loop, drawing deterministic text/buttons onto the game surface after (or instead of) the gameplay frame.
- Headless and programmatic entries (`env.py`, `rl/player_env.py`, `recursive_playtest.py`, `agent_play.py`) keep constructing `Mediator`/`GameSession` directly, never meet the controller, and gain no display dependency; the display audit stays main-path-only.

## Pause-reason model

- `Mediator` gains one internal reason store (`user`, `menu` in this unit) behind the exact public surface: `is_paused` becomes a property whose getter reports "any reason active" as an exact `bool` (review F7 — checkpoint and observation byte-identity require genuine booleans) and whose setter maps boolean writes to setting or clearing the `user` reason, preserving every direct `mediator.is_paused = ...` write site and pin (research R2, R5); `set_paused(self, paused: bool) -> None` keeps its exact signature and semantics as the `user`-reason writer. Bare-host safety (review F1): the store is per-instance and lazily created — the getter treats a missing store as empty, the setter creates it on first write, and a class-level mutable default is explicitly prohibited (it would leak a held `menu` reason across title/restart reconstructions) — so `Mediator.__new__(Mediator)` hosts in the input-contract tests and the GM-03f differential scripts keep working; a red test exercises the getter and setter on a bare instance. New narrow facades `hold_pause_reason(reason)` / `release_pause_reason(reason)` are controller-only entry points validated against the closed reason vocabulary; releasing a non-held reason is a no-op, holding is idempotent.
- The Space toggle stops writing the raw bool and routes through the same `user`-reason path (`input_coordinator.py:366` currently bypasses `set_paused` — research R1); structured `pause`/`resume` actions and `apply_speed_action` likewise touch only the `user` reason, so a speed selection still implicitly clears the user pause but can never clear a menu hold.
- Every existing consumer keeps reading one effective bool: `transition_active` (mediator.py:625), the GameSession clock-reset gate, PassengerFlow's early return, settle's pause gate, speed-button active states, `env.py:236`, both checkpoint projections, and the privileged oracle all observe the derived property (research R3, R6). GM-06d's unconditional reconcile seam and gated drain+settle are unchanged.
- Reasons are process-local runtime state in this unit: no checkpoint, observation, structured-action, or schema surface changes shape or value beyond the existing bool, so the recursive `paused-time-progression` and `paused-state-mutation` oracles and all frozen v1-v5 artifacts remain byte-exact (research R3, R4); serializing reasons is GM-07b's explicitly deferred scope.
- Multi-reason semantics: pausing via Space while the menu is open is unreachable (input not dispatched); opening the menu while user-paused holds both reasons, and resuming from the menu releases only `menu`, leaving the user pause intact — the game stays paused until the user's own toggle clears it.
