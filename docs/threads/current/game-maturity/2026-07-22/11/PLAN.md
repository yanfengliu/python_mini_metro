## D-027 policy (recorded in the parent decision log within this transaction)

- Autosave triggers (review F1/F2 frame-accurate rewrite): on every PLAYING→PAUSE_MENU entry (immediately after the menu reason is held — the review traced all four gesture fields clear at this boundary in every reachable armed state); on `exit_to_title`, saved BEFORE `_close_pause_menu` releases the menu reason so the document is byte-identical to the menu-entry save (F5 — deliberate redundancy); and on the window-close `pygame.QUIT` path in `main`, gated on state: save only when the controller is in `PAUSE_MENU`, or in `PLAYING` with `mediator.is_game_over` false. The controller's `SystemExit` sites save nothing — the TITLE exit would overwrite the real autosave with a pristine fresh-game document, and the two game-over exits are post-delete. Autosave never fires at `GAME_OVER` or `TITLE` in any path.
- Autosave target: the canonical `saves/autosave.json` under `config.save_dir_name`, created on demand; the existing atomic writer guarantees an interrupted write retains the previous valid save.
- Game over deletes the autosave: a finished run cannot be Continued (Mini Metro fidelity; the forgiving alternative is a one-constant future change). Deletion fires at the PLAYING→GAME_OVER promotion for the interactive path (the promotion runs earlier in the same `handle_event` call as the game-over exits, so those sites are delete-safe), AND on every exit surface when `mediator.is_game_over` is true — including main's QUIT branch, closing the review-proven race where the window-X arrives before any promoting event and would otherwise leave the mid-run autosave resurrectable (F1). Deletion is `unlink(missing_ok=True)` with `OSError` swallowed; a failed deletion never crashes or blocks exit.
- Continue: the button renders when the autosave file exists; loadability is proven at click time (F4 wording), a failed load `(ValueError, OSError)` shows the notice and leaves TITLE intact, and a successful Continue releases the `menu` reason (a discard no-op on QUIT-time saves that never held it), preserves a persisted `user` pause verbatim, and enters `PLAYING`.
- Failure policy on every autosave trigger (F3): swallow `(ValueError, OSError)` — a menu-entry save failure still opens the menu, an exit-path failure still exits, and the previous autosave remains intact by the writer's atomicity; a mid-drag QUIT legitimately fails quiescence with `ValueError` and exits unsaved. Saving never blocks play or exit.

## Design

- `src/app_controller.py` gains two injected seams beside `build_game`, both OPTIONAL with inert defaults (F6 — no autosave, Continue unavailable, no notice; every pinned existing call site keeps working unchanged): `build_from(mediator)` (wraps a loaded Mediator exactly as `build_game` does, including `prepare_layout` — which the review verified calls only position updaters and cannot clobber the loader-restored button/blink state, so no contingency is needed) and an `autosave` seam carrying `save(mediator)`, `delete()`, and `peek()`. The controller calls `save` on menu entry and on `exit_to_title` before the menu release; `delete` on the PLAYING→GAME_OVER promotion and on its game-over exit sites; `peek` for Continue visibility; and `build_from` on successful Continue. `main.py` supplies the closures binding `saves/autosave.json` behind a patchable module-level path seam (F7 — loop tests patch it so suites can never touch a real autosave), and its `pygame.QUIT` branch applies the F1 gate: delete when `is_game_over`, save when in `PAUSE_MENU` or un-over `PLAYING`, then exit.
- Continue title control: `title_layout` gains a `continue` key between `new_game` and `exit` with the three coupled `menu_screens` edits made in lockstep (keys tuple, draw iterator, heading anchor stays the first key); the layout function takes the existing signature and the draw function receives whether Continue is available, drawing it only then (rect present in the layout regardless, so hit-testing stays deterministic; an unavailable Continue is simply not drawn and clicks fall through to nothing). A small public text-notice primitive is added to `menu_screens` for the load-failure line, drawn above the buttons on the title screen after a failed Continue click until the next state change.
- Continue load path: on click with an existing file, `load_game` inside a `try` catching `(ValueError, OSError)`; failure sets the notice and stays on TITLE; success releases `menu`, then `build_from(loaded)` replaces the triple and the state enters PLAYING. The research's unverified interaction — `prepare_layout` on a loaded Mediator versus the loader's restored button/blink state — is probed first in the red phase; if `prepare_layout` clobbers restored state, `build_from` re-applies the loader's button restoration after layout (a small ordering fix inside the factory, not a loader change).
- The autosave file's saved pause reasons at the menu boundary include `menu` (and `user` when held); Continue's release of `menu` is the GM-07b-deferred screen reconciliation, matching the determinism-worker precedent.
- Budgets: all edits fit existing files (`app_controller.py` 170, `menu_screens.py` 130, `main.py` 125 — large headroom); no new module is expected; if the autosave closure family grows past taste, a small `src/autosave.py` (< 500) may host it.

## Acceptance

- Escape to the menu writes `saves/autosave.json` atomically, and Exit-to-title rewrites the byte-identical save before leaving the menu; an interrupted write retains the previous valid save, and a save failure never blocks entering the menu or exiting.
- Closing the window preserves the latest menu-boundary save when the game is mid-run (in the pause menu, or in un-over play), and deletes it when the game is over — including a window close at the game-over screen before any promoting event — so a finished run can never be Continued; a mid-drag window close exits unsaved without crashing.
- The title screen shows Continue when `saves/autosave.json` exists; a click resumes the saved game (checkpoint-identical to the saved boundary, `user` pause honored) while a corrupt or unreadable save shows a notice without crashing or leaving TITLE.
- Game over deletes the autosave at the promotion; a finished run cannot be Continued.
- Headless and programmatic surfaces are untouched (isolation scans stay green); no schema, protocol, observation, or frozen-artifact change; all budgets and gates hold.
