## Design

The tutorial is a REAL seeded game with a coaching overlay: a new `AppScreen.TUTORIAL` runs the normal game loop (the player uses the real controls) while a pure step machine observes the live mediator each frame and advances when the player performs the taught action. It is menu-launched, skippable, stateless across runs (no autosave, no high score, no schema/first-run flag), and â€” because `reconcile_game_over` only fires on `PLAYING` â€” never promotes to the GAME_OVER screen.

- The seeded tutorial mediator SUPPRESSES its own game-over (review MAJOR-1, probe-confirmed): `is_game_over` flipping would freeze the sim permanently (`game_session.py:54`, `passenger_flow.py:148`) and soft-lock the time-dependent steps, plus paint "Game Over" chrome (`game_renderer.py:158`). So `build_tutorial` sets `mediator.overdue_passenger_threshold = 10**9` on the instance â€” a per-instance write (verified to keep the sim alive 120 s+ with no input while `is_in_warning_window` still fires at 30 s), NOT a Mediator class or `config` change. This is an owned deviation from "reads the mediator only": the tutorial CONFIGURES its own seeded instance (exactly as it seeds it), touching no other game and no shared state.
- New pure `src/tutorial.py` (< 300, imports only stdlib â€” no gameplay import, so it stays off every headless/RL import graph, and reads mediator attributes only like the audio differ):
  - `TUTORIAL_STEPS`: an ordered tuple, each `(key, prompt, kind, predicate)` where `kind` is `state` (advanced by observing the mediator) or `dwell` (advanced by accumulated unpaused time), and `predicate(current_snapshot, baseline_snapshot) -> bool`. The seven lessons, ORDERED so setup precedes the running lessons:
    1. `draw` (routes) â€” "Drag between two stations to draw your first line." â†’ committed path count (paths where not `is_being_created`) increased. [state]
    2. `train` (fleet resources) â€” "Click a line's train slot to add a train â€” trains carry more passengers." â†’ `len(mediator.metros)` increased (edge). [state]
    3. `deliver` (line purchases/economy) â€” "Watch your train deliver a passenger to its matching shape â€” deliveries earn credits toward new lines." â†’ `mediator.deliveries` increased. (The coached-normal-game approach cannot force an actual line PURCHASE â€” buying line 2 needs ~90 deliveries â€” so line purchases is taught conceptually through the economy that funds it; an owned scoping decision.) [state]
    4. `reroute` (reroutes) â€” "Select a line (tap it), then drag its endpoint to extend or reroute it." â†’ an EXISTING `path.id`'s `(station_ids, looped)` signature changed (review MINOR-4: the prompt tells the first-timer to select first, or the drag starts a new line). [state]
    5. `overload` (overload pressure) â€” "Passengers waiting too long overload a station â€” watch the warning rings and keep lines flowing." â†’ OBSERVATIONAL (review MAJOR-2): auto-advances after `OVERLOAD_DWELL_MS` (â‰ˆ 4-5 s) of UNPAUSED play on the step, OR sooner if a passenger enters the warning window (`max passenger wait_ms >= WARNING_START_MS`, i.e. 30 s). A good player never overloads, so a pure crowd/wait gate would soft-lock; the dwell guarantees completion while the warning-window OR-term keeps it honest. [dwell]
    6. `pause` (pause) â€” "Press Space to pause and plan." â†’ `mediator.is_paused` is currently true (CURRENT state, not an edge â€” review MINOR-3 analog: if the player already paused during a wait, the step is satisfied rather than soft-locked). In TUTORIAL only the user reason is ever held (Escape skips instead of opening the pause menu), so `is_paused` is the user pause. [state]
    7. `speed` (speeds) â€” "Press 2 or 3 to speed up time." â†’ `mediator.game_speed_multiplier != 1` (CURRENT state, not an edge â€” review MINOR-3: auto-satisfied if the player already sped up to skip a wait). [state]
  - `tutorial_snapshot(mediator) -> dict`: a duck-typed, TOLERANT capture (every field via `getattr`, absent â†’ inert) of `{committed_paths, route_signatures, metros, deliveries, paused, speed, max_wait_ms}`, copying route tuples out of the live lists (reroute mutates `path.stations[:]` in place). Tolerance keeps a partial/mocked host from ever crashing the loop, like `audio.snapshot_of`.
  - `TutorialProgress` (frozen): `(index, baseline, dwell_ms, done)`. `start_progress(mediator)` captures step 0's baseline; `advance(progress, mediator, elapsed_ms, paused) -> progress` â€” for a `state` step evaluates the predicate against the live snapshot; for a `dwell` step accumulates `elapsed_ms` into `dwell_ms` only while not paused and completes at the threshold (or the state OR-term); on completion advances the index, re-baselines, and resets `dwell_ms` (so each step diffs against the moment it activated); at the last step sets `done`. Pure and idempotent.
  - `current_prompt(progress)`, `step_ordinal(progress)`/`STEP_TOTAL`, `is_complete(progress)` for the overlay.
- `src/app_controller.py`: a new `AppScreen.TUTORIAL`; an optional inert `build_tutorial` seam (a zero-arg callable returning the seeded, game-over-suppressed triple, like `build_from`); `self._tutorial_progress`; `_start_tutorial()` (build the triple through the seam, set `TUTORIAL`, capture the initial progress â€” no autosave, no highscore); `_handle_tutorial(event)` (Escape â†’ dispatch the letterbox-cancel `MouseEvent(MOUSE_UP, Point(-1,-1))` then return to TITLE, mirroring `_handle_playing` (review NIT-5); otherwise dispatch to the live session so the REAL controls drive the game); a public idempotent `advance_tutorial(elapsed_ms)` (a no-op unless `TUTORIAL`; observes the mediator and advances the step, mirroring `reconcile_game_over`); and the title-branch wiring for the new `tutorial` control. Seam-less (no `build_tutorial`) the Tutorial entry is inert, exactly as Continue is without an autosave.
- `src/main.py`: build a `build_tutorial` closure (`Mediator(seed=TUTORIAL_SEED)`, set the game-over-suppression attribute, a fresh renderer/session, `prepare_layout`) and pass it to the controller; `TUTORIAL_SEED = 42` (the one probed seed whose initial three stations are all DISTINCT shapes, so any line the player draws enables deliveries); a per-frame `controller.advance_tutorial(elapsed_ms)` call beside `reconcile_game_over` (reads post-advance mediator state so tick-driven deliveries register, and feeds the dwell timer); and a `TUTORIAL` render branch that draws the real game frame then `draw_tutorial_overlay(...)` above it. The existing advance gate already advances real time for a non-TITLE/SETTINGS state, so the tutorial game ticks (and freezes on user pause). Audio stays gated to PLAYING/PAUSE_MENU/GAME_OVER (the tutorial is silent this unit â€” a deliberate deferral, not a change).
- `src/ui/menu_screens.py`: a `tutorial` entry APPENDED last to `title_layout` (`("new_game","continue","exit","settings","tutorial")`), so the existing four rects stay byte-identical (the GM-07c/GM-08a append pattern; no test pins `len(title_layout)` or a golden title image â€” review-confirmed) and their tests stay green; `draw_title_screen` paints the fifth "Tutorial" button; and a byte-stable `draw_tutorial_overlay(surface, prompt, ordinal, total, done)` that paints a coaching banner (prompt + "Step N/7" + an "Esc to skip" hint; on completion, "Tutorial complete â€” Esc to return"). ## Isolation and invariants

- `tutorial.py` imports no gameplay/rendering/save module (reads mediator attributes only), and only `app_controller`/`main` import it â€” both already human-shell-only and outside every isolation-scanned surface (`env.py`, `agent_play.py`, `recursive_*`, `rl/*`). A red isolation test adds `"tutorial"` to the GM-08a scan's `forbidden` set (review-confirmed consistent) so no headless/RL surface imports it; it is NOT a persistence module, so it does not join `SAVE_MODULE_NAMES`.
- No `Mediator`/`GameSession`/`rendering` CLASS change and no `config` change (the `TUTORIAL_SEED` and the game-over-suppression value live in `main`, not `config`); no save schema, observation, action, checkpoint, protocol, or frozen-artifact change; settings untouched (no first-run flag â€” menu-launched). The ONE owned deviation from "reads only" is the per-instance `mediator.overdue_passenger_threshold = 10**9` write in `build_tutorial` (review MAJOR-1): it configures the tutorial's OWN seeded instance at construction, exactly as `seed=42` does, and touches no class, no config, no other game, and no shared state â€” the mediator's code is unchanged.
- The tutorial never promotes game over (TUTORIAL â‰  PLAYING) AND its instance never reaches game over at all (the suppression keeps the sim alive), never autosaves, never records a high score, and Escape always returns to the title â€” a self-contained coached experience layered over an otherwise-unchanged game.
