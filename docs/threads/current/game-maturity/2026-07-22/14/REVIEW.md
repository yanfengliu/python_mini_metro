# GM-07b:E review synthesis — locomotive assign/queue/cancel stale-sibling tolerance

## Change

The mutation-path locomotive twin of the GM-07b:D carriage fix. The same reachable stale-but-structural `_station_service_action` (a later same-station metro consumes an earlier metro's cached boarding rider inside one tick; the passenger stays live; GM-07b save/load persists it verbatim) made the strict `_queue_state_is_canonical` return False, so `can_assign_locomotive`/`assign_locomotive`, `queue_locomotive_unassignment`, `cancel_unassignment`/`can_cancel_unassignment`, and the public `queued_locomotives_for_path` count silently no-opped (or read a spurious 0) on every path during the one-tick self-healing window — even with free locomotives or a genuinely queued metro. Seed-9 (`test_gm06c_carriage_stale_sibling._reach_stale_window`) reproduces it publicly. The fix threads the pre-existing opt-in `allow_stale_bound` through the fleet guards (new `_fleet_state_is_canonical` helper at the five read/precondition/postcondition sites; an `allow_stale_bound` parameter on `_detach` for the queue fast path and on `reconcile_queue_transition` for the queue/cancel at-station rebind), keeping the touched metro's own cache strictly oracle-bound and `settle`/path-lifecycle removal strict.

## Lanes

Two independent adversarial lanes ran against the committed diff (`ab91ca4`), each told to read the live code and refute the transactional-safety claims.

- **Harness lane** (`raw/harness-review.md`, **CLEAN**): read every relevant module end-to-end and ran five executable probes against the reachable seed-9 window. All seven refutation targets held by execution — the touched metro stays strictly oracle-canonical after every committed op; an unrelated stale sibling is preserved by object identity on commit; sibling + touched flag/cache/timers are restored verbatim on forced rollback (probed cancel-reconcile, queue-reconcile, and assign failures); `settle` returns 0 and `remove_path` no-ops in the window (the relaxation did not leak); and — the highest-value target — all eight genuinely-corrupt cache shapes (unknown kind, off-invariant timers, bound-off-station, null-with-timers, dangling/removed passenger) are STILL rejected by the relaxed gate and the public ops. It confirmed `queued_count` feeds only the test-consumed `queued_locomotives_for_path`, the UI badge counts raw flags independently, and the checkpoint reads the raw `is_unassignment_queued` flag — so no checkpoint/save value changes. Two NIT observations, both pre-existing and non-blocking (the fast path discards `_detach`'s bool return — byte-identical to `HEAD~1`; `allow_stale_bound` per-metro tolerance is uncapped by design, identical to the GM-07b:C checkpoint contract, probe-confirmed non-harmful).

- **Codex external lane** (`gpt-5.6-sol`, `raw/codex.md`, **NOT CLEAN — 2 P2**). Both findings verified against the live code before acting:

  - **P2 (assign effectful-factory).** `assign` snapshotted only its two owner collections and rolled back only those, relying on the injected factory being a pure constructor to leave unrelated state untouched; an effectful factory that rebinds a sibling's cache to another structurally-valid, still-live action (`('board', p)` → `('transfer', p)`) could commit with the sibling modified, and if the factory raised, the modified cache would remain. Verified: unreachable through the public seam (`mediator.assign_locomotive` hard-wires `get_metro_factory=lambda: Metro`, and `Metro.__init__`/`Path.add_metro` are pure wrt siblings and host RNG — so the reachable guarantee held and was already pinned by a test). But GM-07b:E is exactly what newly lets `assign` commit while a stale sibling is present, the user's own task directive called for `transaction_state_matches`-identity sibling preservation per op, and the carriage/queue/cancel twins already snapshot. **Disposition — fixed.** `assign` now mirrors carriage `attach`: `snapshot_transaction_state` before the factory, `transaction_state_matches(host, state)` after it, `transaction_state_matches(host, state, added_owner=(path, metro))` after the append, and `restore_transaction_state` on any failure. Added a small symmetric `added_owner` mode to `transaction_state_matches` (mirrors `removed_owner`; default None, so attach/detach/reconcile are unchanged). Removed the now-dead `_is_exact_append`/`_restore_owner_lists`/`_restore_collection` helpers. Two red-first regressions: an effectful factory that rebinds the sibling is rejected and the sibling restored verbatim; a raising factory rolls back the whole fleet + sibling verbatim.

  - **P2 (observation/badge claim accuracy).** The commit message / PROGRESS claim "no observation bytes change" was too broad and the `queued_count` comment mislabeled its consumer. Verified: `ui/fleet_button.py:63-75` (`_is_enabled`) reads the relaxed `can_assign_locomotive`/`can_queue_locomotive_unassignment`, and `rl/player_env.py:264` renders the full game (buttons included) into pixels — so the pixel observation legitimately differs during the window (controls become active), which is the intended effect of the fix, not a regression; and successful actions change raw metro/queue state the structured observation serializes. Meanwhile `fleet_button._queued_count` (`:77-88`) counts the raw `is_unassignment_queued` flag directly and never calls the relaxed facade, so `queued_count` affects only the public `queued_locomotives_for_path` (test/programmatic). **Disposition — fixed (accuracy).** Narrowed the PROGRESS bullet to "no schema/serialization change; frozen fixtures byte-exact; the fix intentionally activates the controls (rendered color + post-action observation) in the window; the badge counts raw flags and is unaffected," corrected the `queued_count` comment, and added a public-count assertion (`queued_locomotives_for_path == 1` in the window before cancel). No code-behavior change was warranted — the pixel/structured differences are the fix working as intended.
